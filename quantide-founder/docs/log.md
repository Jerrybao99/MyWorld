# 任务 1 开发需求挖掘
## 用户故事
作为一名写作者，当我完成一段或一篇内容后，我希望可以通过命令行触发 AI 自动扫描文本，并在合适的位置添加结构化批注，而不修改我的原文内容。这样我既可以：保留完整的写作思考过程，又能获得 AI 的额外洞察与总结

## 核心需求
实现嵌入式批注  

+ AI 不修改原文  
+ AI 不重写内容  
+ AI 仅在用户写作完成后，对段落进行扫描和标注  
+ 通过结构化批注提供观察者视角的洞察  

保留用户完整的思维过程   

提供 AI 的结构化分析  

保证批注过程可重复执行（幂等）

# 任务 2 开发流程设计
## 核心工作流程
系统在用户完成写作或写作暂停时触发，对全文进行自动分析与批注生成。整体流程如下：

1. 手动触发：python -m thera.cli 
2. 调用 llm 工具：deepseek
3. 网络异常：自动重试 3 次；仍失败则返回错误并停止处理
4. 解析异常：跳过当前段落，记录日志，继续处理后续段落
5. 全文扫描：在写作完成或检测到写作停顿时，系统读取当前文档内容
6. 段落切分：按 \n\n（空行）将全文切分为多个逻辑段落，每个段落视为独立分析单元
7. 段落过滤：过滤不需处理的段落，包括：
    - 仅包含标点的段落
    - 代码块内容
    - 长度小于 20 字的段落
    - 无意义闲聊
8. 状态检测：检查段落是否已被处理或已存在批注，避免重复分析
    - 检测方式：查找是否存在标记 🤖 观察者注
    - 检测规则：
        * 若段落已存在该标记 → 跳过处理（保持幂等性）
        * 若段落不存在该标记 → 标记为 待观察段落
9. 内容分析：对包含洞察、决策或风险相关信息的段落调用 AI 进行提炼分析
    - 若段落仅为闲聊或无信息价值，输出：SKIP
10. 生成批注：若段落包含有效信息，AI 以“观察者视角”生成结构化批注，不修改不重写原始文本内容，AI 批注示例，且 AI 批注皆使用引用语法 >： 

    > 🤖 观察者注：处理后段落，检测标识  
    >
    > 🏷️ 标签：#工作优化 #成本管理 #AI协作 #未来工作 #金字塔模型  
    >
    > 💎 提炼：提取的标准化知识。工作日志自然化记录，利用低成本模型处理大量tokens，构建金字塔型工作流提升创意层级  
    >
    > ⚠️ 状态：状态说明。包含对工作方式优化的深度洞察和未来工作形态的展望（可选）  
    >
    > 🔗 关联：关联说明。AI协作、成本优化、工作流程重构（可选）  
    >
    > 🔑 关键：关键要点。自然记录、成本优先、金字塔结构、创意升级（可选）  
    >
    > 🔄 模式：模式说明。自下而上处理简单任务，自上而下专注创意工作（可选）
    >

11. 合成回写：将原始段落与 AI 生成的批注进行拼接，并写回文档，保留完整写作过程，同时补充 AI 提供的洞察信息

## 基本开发流程
目录创建 → 环境文件配置/`config.py` → `llm.py` → `mode/default.py` → `cli.py` → 测试

# 任务 4 开发平台功能设计
## 设计
### 模块设计
原则：分层架构，单一职责，最小可用

`src/thera/cli.py`：命令行入口，越薄越好，胶水代码  

`src/thera/config.py`：集中管理配置项，配置集中管理：Settings 类集中定义所有配置  

`src/thera/llm.py`：封装与 deepseek 的交互，隔离外部依赖，便于替换  

`src/thera/mode/default.py`：协调各模块、处理业务逻辑，编排而非实现，把控整体流程

### 函数划分
| 函数名 | 功能 |
| --- | --- |
| `split_paragraphs` | 段落切分，并过滤空字符串和过短段落 |
| `is_already_annotated` | 检测段落是否已经处理，避免重复批注 |
| `is_short` | 判断段落是否过短（小于 20 字） |
| `get_annotation` | 调用 LLM 获取文本批注 |
| `load_config` | 从 YAML 文件加载配置 |
| `read_file` | 读取文件内容 |
| `write_file` | 写入文件内容 |




# 任务 5 代码与测试规范设计
## 代码规范
### 基本标准
遵循最佳工程实践  

避免重复和冗余代码（DRY）  

保持模块职责单一  

代码应清晰可维护  

### 命名管理
显式标注：关键信息明写而不依赖默认或隐含理解  

领域驱动，见名知意，简短清晰  

全英小写（snake_case），必要复数  

| 类型 | 规范 | 示例 |
| --- | --- | --- |
| 文件名 | snake_case | config_loader.py |
| 变量名 | snake_case | user_name |
| 函数名 | snake_case | load_config() |
| 类名 | PascalCase | UserService |
| 常量 | 全大写 + 下划线 | MAX_RETRY |
| 模块名 | 简短且语义清晰 | llm, config, cli |


### 配置管理
参数抽离：路径、超时、模型参数等必须抽离为配置或参数  

禁止硬编码：config.py、.env、pyproject.toml 等配置文件管理 API Key、文件路径、URL 地址、业务参数、环境配置  

配置集中管理：Settings 类集中定义所有配置  

统一访问：业务代码通过统一的配置对象读取参数  

### 依赖管理
导入依赖：明确在代码开头写出需要用到的库，例如 import requests。  

外部依赖：防止用户环境没有安装库  

```python
try:
    import requests
except ImportError:
    print("请先安装 requests")
```

隐式依赖：必须声明，代码内部用到的库也要写在依赖列表里  

> requirements.txt 中写 numpy  
>

重复导入优化为缓存：统一在模块入口导入，同一个模块多次导入时复用缓存，避免反复加载

## 测试规范
### 覆盖规则
**一区一测**：每个功能区域/文件必须至少有一个测试

**Mode 必测**：不同运行模式必须分别测试

### 测试流程
导入测试 → 配置测试 → 依赖测试 → 单元测试 → 集成测试

| 问题 | 位置 | 说明 |
